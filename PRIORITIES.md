## Brainstormed & shipped (2026-07-03 afternoon)

- **Phase 3 dashboard cache decoupling** — The whole `PrizePicksClient` was wrapped in a single `tokio::sync::Mutex<Arc<...>>` (`PrizePicksState`). Every Tauri command that touched the cache — reads *and* writes — had to `lock().await` the whole client. The full-catalog warm (20 pages × `MAX_PAGINATION_PAGES`, ~10s+ of HTTP round-trips) held the same mutex, so during a `prizepicks_refresh` or the 8s startup background warm, every read command (`prizepicks_get_top_markets`, `prizepicks_search_markets`, `prizepicks_get_category_stats`, `prizepicks_get_cache_status`, the dashboard bootstrap) blocked for the duration of the warm. Two concurrent fetches (e.g. an explicit refresh + the 8s startup warm) would also each run a full 20-page sweep, clobbering each other. The Phase 3 roadmap item called for an `Arc<RwLock<...>>` + fetch guard; this pass shipped it. **The headline win**: a user clicking "Refresh" no longer freezes the dashboard for 10 seconds — the UI keeps reading the existing quick cache while the warm runs in the background. Shipped:
  - `src-tauri/src/prizepicks/client.rs` — `PrizePicksClient.cache: Option<PrizePicksCache>` → `cache: Arc<RwLock<Option<PrizePicksCache>>>` (many concurrent readers, single writer). New `fetch_in_progress: Arc<AtomicBool>` fetch guard (prevents two concurrent 20-page sweeps). `is_cache_stale` / `get_cached` / `needs_full_catalog` / `cache_status` / `category_stats` all take a read-lock (cheap, many-concurrent). `store_cache` / `invalidate_cache` take a write-lock (briefly, only to swap the inner `Option`). `ensure_quick_cache` and `fetch_all_markets` take the read-lock only to short-circuit on a fresh full cache, run the HTTP loop WITHOUT holding the lock at all, then take the write-lock to install the result. New `try_begin_fetch` / `end_fetch` / `wait_for_in_flight_fetch` helpers; `wait_for_in_flight_fetch` polls the cache for fresh full-catalog status with a 30s safety net so a stuck fetch doesn't deadlock the caller.
