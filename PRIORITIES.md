## 2026-07-05 afternoon pass #3 — Per-command correlation_id (fc272a9)

**Feature shipped (per-command `correlation_id` for trace grouping):** Yesterday's structured-logging-foundation pass gave us a working `tracing_subscriber` (human + JSON modes, auto-ANSI suppression) and a parallel frontend `logger.ts` wrapper. The next smallest step toward the Phase 5 OpenTelemetry observability item is giving every Tauri command a stable per-invocation id, so all log lines emitted during one user action can be grouped by `grep <cid>` (or `jq '.fields.correlation_id'` in the JSON sink). This is the pre-OTel "trace id" pattern, implemented without a single new dependency. Shipped:

- `src-tauri/src/logging.rs` — new `pub fn new_correlation_id() -> String` that returns an 8-char lowercase-hex string derived from the bottom 32 bits of the `UNIX_EPOCH` nanosecond count. 8 hex chars = 32 bits of entropy — enough to keep collisions vanishingly rare across a single user session (the rapid-call test asserts ≥990/1000 distinct values), and short enough to type from a screenshot / grep through 5k log lines without scrolling. **Why not `Uuid::new_v4()`**: pulling in the `uuid` crate is a heavy dependency for a value that's only used as a log key. The nanosecond clock is already in `std` and is sufficient for the "group a single command's log lines together" use case. **Caveat documented in the docstring**: this is NOT a W3C `trace_id` (those are 16 bytes / 32 hex chars). The full OTel `trace_id` + `span_id` pair will arrive when the project adopts an OTel SDK; the cid is the pre-OTel stepping stone that lets us group log lines by user action today. 3 unit tests cover the format invariants: `correlation_id_is_8_char_lowercase_hex` (length = 8, all chars are ASCII lowercase hex), `correlation_id_changes_across_rapid_calls` (1000 rapid calls produce ≥990 distinct values, tolerating platform-specific nanosecond resolution), and `correlation_id_uses_only_hex_alphabet` (50 calls × 8 chars = 400 chars, all `0-9a-f` — pins the alphabet so a future change to base36 fails loudly). A 4th test, `correlation_id_survives_into_tracking_event_field`, is an end-to-end subscriber test that wires a real `fmt::layer().with_writer(...)` pointing at a `BufferWriter(Arc<Mutex<Vec<u8>>>)`, emits a `tracing::info!` event with `correlation_id = %cid`, and asserts the captured stream contains both the cid and the event message. The `%cid` (Display) formatter is what makes the field `jq`/`grep`-friendly — switching to `cid` without the `%` would emit the field's `Debug` repr, which a downstream `jq` filter would not match. 152 lines added.
- `src-tauri/src/commands/prizepicks_cmd.rs` — two highest-traffic commands instrumented: `prizepicks_get_dashboard_bootstrap` (the single IPC round-trip that fires on every dashboard mount) and `prizepicks_refresh` (the full-catalog warm, the most common user-reported hang). Both generate a cid at function entry and emit 2-3 `tracing::info!` / `tracing::warn!` events with `correlation_id = %cid` as a structured field. Dashboard bootstrap: 1 start event with `limit` and the cid, 1 end event with `props_count` / `scored_props_count` / `cache_full` and the cid. Refresh: 1 start event (the moment a user clicks Refresh), 1 warn event on snapshot failure (carries the cid so the failure is traceable to the click that triggered it), 1 end event with `markets_count` / `summaries_count`. The `props` binding in bootstrap gained a `: Vec<_>` type annotation (previously inferred only at the `Ok(...)` return) because the new `tracing::info!(props_count = props.len(), ...)` macro requires the type to be known at that call site. 46 lines added.

- **Phase 3 dashboard cache decoupling** — The whole `PrizePicksClient` was wrapped in a single `tokio::sync::Mutex<Arc<...>>` (`PrizePicksState`). Every Tauri command that touched the cache — reads *and* writes — had to `lock().await` the whole client. The full-catalog warm (20 pages × `MAX_PAGINATION_PAGES`, ~10s+ of HTTP round-trips) held the same mutex, so during a `prizepicks_refresh` or the 8s startup background warm, every read command (`prizepicks_get_top_markets`, `prizepicks_search_markets`, `prizepicks_get_category_stats`, `prizepicks_get_cache_status`, the dashboard bootstrap) blocked for the duration of the warm. The Phase 3 roadmap item called for an `Arc<RwLock<...>>` + fetch guard; this pass shipped it. **The headline win**: a user clicking "Refresh" no longer freezes the dashboard for 10 seconds — the UI keeps reading the existing quick cache while the warm runs in the background. Shipped:
  - `src-tauri/src/prizepicks/client.rs` — `PrizePicksClient.cache: Option<PrizePicksCache>` → `cache: Arc<RwLock<Option<PrizePicksCache>>>` (many concurrent readers, single writer). New `try_begin_fetch` / `end_fetch` / `wait_for_in_flight_fetch` helpers; `wait_for_in_flight_fetch` polls the cache for fresh full-catalog status with a 30s safety net so a stuck fetch doesn't deadlock the caller.
